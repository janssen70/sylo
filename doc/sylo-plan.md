# Self-Contained Syslog Recorder — Project Plan

## Scale targets (derived)
- Devices: 100 nominal, design for 1,000 (10x headroom)
- Volume: ~300/device/day nominal → up to ~300,000 msgs/day at 10x
- Retention: 1 year default, configurable → up to ~110M rows/year at ceiling
- Implication: SQLite is fine for this volume with proper indexing, but plan for monthly file/table partitioning so no single file grows unbounded and old partitions can be dropped in O(1) instead of deleted row-by-row.

## Deployment
- v1 ships localhost-only (bind 127.0.0.1)
- Design decisions (auth, config, HTTP layer) must not preclude later LAN binding + reverse proxy in front for cloud/scale deployment.
- Primary distribution target is the Windows installer (section 5). Core code (receiver, indexer, webapp, retention) is plain asyncio/SQLite with no Windows-specific imports outside `sylo/receiver/winservice.py`, so it already runs unmodified on Linux -- section 8 covers what's needed to make that a properly supported deployment (service supervision, the privileged-port permission difference) rather than just "happens to work."

## 1. Receiver (syslog listener) spec
- [x] Protocol support: UDP 514 (primary), TCP 514 (reliable option, newline-delimited framing), both RFC3164 and RFC5424 parsing supported, malformed input tolerated (never raises, falls back to raw message + `malformed` flag)
- [x] Concurrency model: **single asyncio event loop** for UDP/TCP ingest + small shared bounded thread pool (4-8 threads via `run_in_executor`) for actual file write/fsync calls. Event loop callback only parses device ID and enqueues — never touches disk directly.
  - Per-device isolation: one file + one `asyncio.Queue` + one writer coroutine per device, **keyed by source IP address** (not the parsed hostname field — a malformed/spoofed packet must not be able to inject an arbitrary device identity or path-unsafe characters into the file/queue key); a slow/stalled write for device X cannot block device Y (only shares the small executor pool, not the event loop).
  - Flush policy: writer flushes on buffer size/count threshold OR idle timer (~1s) elapse, whichever first — balances write batching against read freshness. fsync on a separate, slightly longer interval (~2s).
  - Overload detection: bounded per-device queue; soft threshold logs a warning + increments a lag counter, hard cap drops messages for that device only (never blocks ingest) + increments a drop counter. Counters exposed via health/metrics endpoint.
- [x] Message envelope: capture receipt timestamp, source IP, raw bytes, parsed fields (facility, severity, host, tag, msg)
- [x] Write path: append-only to **daily** rotating text files, one file per device per day (independent of the monthly SQLite partitioning in section 2 — a folder-per-day layout stays simple even if retention grows well past a year). fsync policy: see flush policy above (~2s interval).
- [x] Backpressure/overload behavior defined (what happens if disk or indexer stalls — must not block socket accept)
- [x] Runs as independent Windows service, no dependency on UI process: `sylo/receiver/winservice.py` wraps `receiver.main.run` in a pywin32 `ServiceFramework` (`SyloReceiverService`, `python -m sylo.receiver.winservice install|start|stop|remove`). Only imports from `sylo.receiver` -- no import of `sylo.webapp`, so install/start/stop/restart of this service never touches the UI service. `run()` now takes an optional external `stop_event`; `SvcStop` runs on a thread handed to it by the Service Control Manager, separate from the thread running the event loop, so it signals shutdown via `loop.call_soon_threadsafe(stop_event.set)` rather than setting the event directly. pywin32 is Windows-only (`sys_platform == 'win32'` marker in pyproject.toml) and isn't installed/testable in this Linux dev environment -- verified here only via the stop-event cross-thread plumbing (`tests/test_winservice.py`); the actual service registration/lifecycle needs verification on a Windows box.
- [x] Crash/restart behavior: safe resume, no data loss/corruption of in-progress file. Per-device day files are reopened in append mode on demand, so a plain process restart resumes writing to the same file with no special recovery step. The one gap that needed code: a hard crash between `write()` and the next `fsync` can leave the file's last line torn (no trailing newline) -- `write()` itself is all-or-nothing, but the underlying dirty page may only be partially flushed to disk when power is lost. Appending straight onto that torn tail would silently merge a brand-new, otherwise-intact message onto the old garbage, corrupting it too. Fixed via `_ensure_clean_tail` (`sylo/receiver/device_writer.py`): before the first write to a given day-file in a process's lifetime, if the file's last byte isn't `\n`, one gets appended first, isolating any stale torn fragment onto its own line so new messages are never merged into it (`DeviceWriter._sanitized_paths` tracks which paths have already been checked, so this runs once per file, not on every flush). The stray fragment itself is left as-is (truncated messages right at the crash boundary are an accepted, already-documented loss window); the important guarantee is that it can never corrupt subsequent messages. Messages still queued in memory (not yet flushed) at crash time are lost, same as the existing documented drop/backpressure behavior -- unavoidable without WAL-style write-ahead logging on the raw text path, which was judged unnecessary complexity for this scale. SQLite's own crash safety (WAL mode) already covers the index; if the index and raw files diverge after a crash, `indexer.rebuild` (line 37) reconciles them. Covered by `tests/test_device_writer.py::test_restart_after_crash_does_not_merge_onto_torn_line`.

## 2. Storage / indexer spec
- [x] Source-of-truth: rotating plain text files (open format, always readable even if index breaks)
- [x] Index: SQLite (WAL mode), schema with indexes on (timestamp), (host, timestamp), (severity, timestamp), (facility, timestamp) -- composite indexes chosen over single-column ones since section 3's filters always pair a dimension with a time range/order. Also an external-content FTS5 table (`messages_fts`) over message/host/tag for section 3's free-text filter -- a plain `LIKE '%term%'` scan wasn't going to stay interactive anywhere near the 110M-row/year ceiling. Kept in sync on insert only (no update/delete triggers), since retention only ever drops a whole month's DB file, never individual rows.
- [x] Partitioning strategy: one DB/table-set per month to bound size and simplify retention drops
- [x] Indexer placement: **embedded in receiver process** (own asyncio task/queue, not the UI process). Chosen over separate process — write path is expected to stabilize quickly and change rarely, while search/lookup will see considerable rework, so isolating "likely to change" (indexer/UI) from "rarely changes" (receiver) matters more here than a hard process boundary.
  - Safety nets to preserve ingest isolation despite sharing a process:
    - Each message's parse+insert wrapped in its own try/except — malformed input must never raise into the ingest loop
    - Indexer has its own queue + task, same pattern as per-device writers — slow SQLite operations delay indexing only, never ingest
    - Batched commits (N rows or T seconds) to bound lock/fsync overhead per operation
    - Lag/backlog counter for the indexer (mirrors the write-side overload counter) exposed via health endpoint
- [x] Recovery: indexer can rebuild index from text files if SQLite is lost/corrupted (`python -m sylo.indexer.rebuild --data-dir ... --index-dir ...`, optionally scoped to specific `--month`s; deletes and recreates the targeted month DB(s) before reinserting)

## 3. HTTP / UI spec
- [x] Server: FastAPI + uvicorn (Python), single process, separate from receiver (`sylo/webapp`, run via `python -m sylo.webapp.main`)
- [x] Bind: 127.0.0.1 only in v1; config flag reserved for future LAN bind (`SYLO_WEB_BIND_HOST`)
- [x] API: paginated `/api/messages` with filters (host, severity, facility, time range, free-text via FTS5). Pagination spans multiple monthly DB files by querying each for its own top (offset+limit+1) matches, merging, and re-sorting -- exact for the page returned; bounded to a configurable number of recent months when no explicit time range is given, to avoid an unbounded full-history scan by default.
- [x] Live tail: **SSE** chosen over WebSocket (one-directional data, simpler than a WS handshake, and htmx has a matching first-party SSE extension). Implemented as a polling loop (configurable interval, default 1.5s) over the current month's index DB rather than any cross-process pub/sub -- keeps the read path's "only touches SQLite" property intact.
- [x] Frontend: server-rendered + htmx (no SPA build step); htmx + its SSE extension vendored locally under `sylo/webapp/static/` (no external runtime/CDN dependency, consistent with section 5's packaging goal)
- [x] Read path only touches SQLite index, never raw files, never blocks receiver (webapp is a fully separate process; each query opens its own short-lived read connection to the relevant monthly DB file(s))
- [x] `/healthz` endpoint for supervisor/auto-restart (no auth required, deliberately cheap -- no DB/receiver checks)
- [x] Pages: message browser/search (`/messages`, htmx-paginated), live tail (SSE panel on the same page), retention settings (`/settings/retention`), device/source list (`/devices`)

## 4. Retention manager spec
- [x] Default: 1 year, user-configurable (per-install setting, UI-editable) -- already in place from section 3/6: `retention_days` lives in the control-plane `settings` table (default `365`, seeded by `appdb.init_db`), editable via `/settings/retention`. The retention manager (below) reads this same setting fresh on every pass rather than caching it, since it can change at any time via the UI.
- [x] Deletion granularity: drop whole monthly DB partitions; for raw text files (daily, per device — see receiver spec) delete all daily files whose date falls inside the dropped month, rather than row-by-row. New `sylo/retention` package: `core.run_retention` finds index months whose *last* day is older than `today - retention_days`, deletes that month's `{month}.sqlite3` plus its WAL-mode `-wal`/`-shm` sidecars, then deletes every per-device `data/raw/<device>/{month}-*.log` file for that same month (glob on the filename prefix, not a per-row/per-file age check).
- [x] Schedule: periodic background job (daily), independent of receiver and UI: `sylo/retention/main.py`, run via `python -m sylo.retention.main` (own process, default 24h loop via `run_interval_seconds`/`SYLO_RETENTION_INTERVAL_SECONDS`; `--once` for a single pass e.g. from an external scheduler). Only imports `sylo.webapp.appdb` (for the shared settings-table accessor) and touches the on-disk data/index layout -- no dependency on the receiver or webapp processes being up, and (fixed during smoke-testing) it calls the idempotent `appdb.init_db` itself so a fresh install works even if retention's first run happens before the webapp has ever started. Loop follows the same external `stop_event` convention as `receiver.main.run`, for the same reason (a future Windows-service wrapper's `SvcStop` needs to cross threads via `call_soon_threadsafe`).
- [x] Safe-guard: never deletes current/active partition -- enforced twice: the month-selection filter in `_expired_month_keys` skips today's month key outright, and each delete helper (`_delete_index_month`, `_delete_raw_month`) independently asserts it was not handed the current month, as defense in depth against a caller bug. Covered by `tests/test_retention.py` (7 tests: expiry-window math, WAL/SHM sidecar cleanup, raw-file month-prefix matching, the zero-retention-days edge case, and the defense-in-depth assertions).

## 5. Windows packaging / installer spec
(Renamed from "Packaging / installer spec" -- this section is Windows-specific; Linux deployment is section 8. This whole section was authored on Linux, with no Windows machine, Wine, or Inno Setup available in this dev environment, so it could only be drafted here, not compiled or run -- actual build/install verification happened later on a real Windows machine, see below.)
- [x] Installer tooling: **Inno Setup** — simplest scripting for bundling multiple exes + installing Windows services + custom install/data paths + uninstall-preserves-data logic; single self-contained setup.exe. (WiX/MSI worth revisiting only if later pushed via enterprise GPO/SCCM.)
- [x] Bundles: receiver service, indexer, UI service (PyInstaller or similar for Python parts), default config -- indexer is embedded in the receiver process (section 2), so there are exactly three exes, one per process: `packaging/pyinstaller/{receiver,webapp,retention}.spec`. Each spec's `Analysis` script is a tiny `entry_*.py` (`from sylo.receiver.winservice import main`, etc.), not `winservice.py` itself -- an actual Windows build attempt caught that freezing a package submodule with relative imports directly as the entry script fails with "attempted relative import with no known parent package" (PyInstaller, like `python winservice.py`, runs the target as a standalone module with no parent package). The indirection through a normal import fixes it while keeping `winservice.py` itself consistent with the rest of the codebase's relative-import style. webapp.spec bundles `sylo/webapp/templates` and `sylo/webapp/static` as datas at matching relative paths, since `app.py` locates them via `Path(__file__).parent` at runtime. Default config (the `SYLO_*` env vars each process needs) is written per-service into the registry by the installer, not shipped as a file -- see the next item. `hiddenimports=["win32timezone"]` is included as a known, commonly-needed pywin32-under-PyInstaller fix; further hidden-import gaps, if any, can only be found by continuing the build past this point on Windows.
- [x] Installs all three processes as Windows services with independent start/stop/restart: extended the receiver's existing `winservice.py` pattern (section 1) to `sylo/webapp/winservice.py` (`SyloWebapp`) and `sylo/retention/winservice.py` (`SyloRetention`) -- required refactoring `webapp.main` to expose an async `run(config, stop_event)` using `uvicorn.Server.serve()` directly instead of the blocking `uvicorn.run()` convenience wrapper, so it can be driven by the same external-stop_event/`call_soon_threadsafe` convention as the receiver and retention services (`tests/test_winservice.py` covers all three, still passing after the refactor -- 82 tests total). `packaging/inno/sylo.iss`'s `[Code]` section registers each service (`<exe> install --startup=auto`), writes that service's `SYLO_*` config directly into its own `HKLM\...\Services\<name>\Environment` registry value (the documented mechanism for per-service env vars, chosen over system-wide env vars so it takes effect immediately with no reboot/logoff), then starts it -- all three are independent units with no `Requires=`/dependency between them, so `sc stop/start SyloWebapp` never touches the other two. Build/run-verified on a real Windows machine: `iscc packaging\inno\sylo.iss` compiles clean and a full install registers, configures, and starts all three services successfully. Getting there past the draft surfaced two real Pascal Script/pywin32 bugs, both fixed in `sylo.iss`: (1) `RegWriteMultiStringValue`'s `Data` parameter is a single null-joined `String`, not an array as the doc's parameter name suggested -- `InstallService` now joins `EnvLines` into a `#0`-delimited string instead of forwarding an array; (2) pywin32's `HandleCommandLine` parses argv with plain `getopt.getopt` (not `gnu_getopt`), so `--startup=auto` must precede the `install` command, not follow it, or it's left as a stray positional argument and the whole call exits 1 -- was `install --startup=auto`, now `--startup=auto install`.
- [ ] Default install/data paths, upgrade/uninstall behavior (data retained on uninstall unless user opts out) -- addressed in `sylo.iss`: binaries under `{autopf}\Sylo`, data under `{commonappdata}\Sylo\data` (`raw`, `index`, `app.sqlite3`), a fixed `AppId` GUID so re-running the installer upgrades in place rather than side-installing, and `[UninstallRun]` stops+removes all three services before file deletion (each entry given a `RunOnceId` so it fires once per version rather than re-running on every in-place-upgrade install, not just from the real uninstaller), followed by a keep-or-delete prompt for the data directory (defaulting to keep, per plan). Install paths verified on a real Windows machine (previous item). Uninstall verified partially: `sylo-receiver.exe`/`sylo-retention.exe` were stopped, unregistered, and deleted correctly, but `sylo-webapp.exe` stayed locked on disk -- root cause was a real bug, not an installer issue: uvicorn's graceful shutdown (`sylo/webapp/main.py`) waited indefinitely for a still-open live-tail SSE connection (a browser tab left on that page) to close on its own before the process could exit, so `stop` returned before the process actually had. Fixed by adding `timeout_graceful_shutdown=5` to the uvicorn `Config`. Still open, not yet exercised: re-running the uninstall test with a browser left on the live-tail page to confirm the fix holds, and the keep-vs-delete data prompt itself -- see section 7's "Installer test: clean Windows VM" item.

  The upgrade cycle specifically *was* exercised (real deployment, pushed the 8514/shortcuts change from the previous pass over an already-installed, already-running Sylo) and failed: Setup couldn't overwrite `sylo-webapp.exe` because the old `SyloWebapp` service still had it open -- Windows can't overwrite a locked exe, upgrade or not, and nothing in the script stopped the old services before `[Files]` ran (`[UninstallRun]`'s stop+remove entries are keyed to actual uninstall/RunOnceId-forwarded-uninstall processing, not to `[Files]` timing specifically, and evidently weren't enough here on their own). Fixed with `PrepareToInstall` (`sylo.iss`) -- the documented Inno hook that runs once, right after the wizard finishes and right before `[Files]` starts copying: it checks whether any of the three services are already registered (`ServiceInstalled`, a `RegKeyExists` check against each service's registry key -- reliable specifically because that key only ever gets created by this same installer's own `InstallService` call, so its presence *is* "previously installed"), and if so, asks the user to confirm before stopping them (`MsgBox`, `MB_YESNO or MB_DEFBUTTON2` -- unlike the uninstall data-keep prompt, "No" is the safer default here since stopping is the disruptive action, not confirming). Declining aborts Setup cleanly via `PrepareToInstall`'s non-empty-string-return convention, rather than proceeding into a doomed file copy. A `Sleep(1000)` cushion after stopping covers the gap between the SCM reporting `SERVICE_STOPPED` and the process actually releasing its exe file handle. Second, related fix from the same report: re-prompting for the admin password and web port on every upgrade was pointless (password is only ever consulted on the webapp's true first run) and risked silently reverting an already-fixed-for-this-machine port back to the shipped default. `IsUpgrade` (same `ServiceInstalled` check, against `SyloWebapp` specifically) now drives `ShouldSkipPage` to skip both wizard pages entirely on upgrade, and `ReadExistingWebPort` reads the port back out of `SyloWebapp`'s existing registry `Environment` value (parsed the same null-separated-string way `InstallService` already writes it) so it carries forward unchanged into the reinstalled service instead of reverting to `{#MyDefaultPort}`. Not yet handled, flagged rather than fixed: there's no way left in the wizard to *change* the port on an upgrade (by design, per the request that skipped this page) -- doing so now needs either a full uninstall/reinstall or a manual registry edit + service restart; and repeated installs keep re-persisting whatever `SYLO_ADMIN_PASSWORD` was set at first install (if any) into `HKLM\...\Services\SyloWebapp\Environment` in plaintext indefinitely, which pre-dates this pass and wasn't in scope to fix here. None of this `[Code]` logic has been run through ISCC yet (same standing limitation as the rest of section 5) -- needs a real Windows upgrade-over-a-running-install cycle to verify `PrepareToInstall`'s confirm/stop/Sleep sequence actually unblocks the file copy, and that the skip/carry-forward logic behaves correctly both on a fresh install (no prior services -- both pages shown, port defaults to 8514) and on upgrade (both pages skipped, port/password behavior as described above).
- [x] No external runtime dependencies required (no separate Python/DB install needed by end user) -- PyInstaller onefile bundles the Python interpreter and all dependencies (FastAPI/uvicorn/Jinja2/bcrypt) into each exe; SQLite is the stdlib `sqlite3` module, bundled the same way, so nothing beyond the three exes needs installing on the target machine.

Known gap carried forward, not solved in this pass: if the admin password wizard page is left blank, `sylo-webapp` auto-generates one and logs it once via the `logging` module -- but a Windows service has no attached console, so that log line currently goes nowhere visible. Fixing this properly needs the webapp to write startup logs to a file or the Windows Event Log, which isn't built. Documented as a known limitation at the top of `sylo.iss` rather than addressed here.

## 6. Auth / security spec
- [x] Auth model: single local admin account (created at install/first-run), backed by a real `users` table (id, username, password_hash, created_at) rather than a hardcoded credential — keeps the door open to adding accounts later without redesign. Password hashed with **bcrypt** (chosen over argon2: simpler pure-Python packaging for PyInstaller, no C-extension build headaches; security difference is not material at single-local-admin scale). Server-side sessions (random token in HTTP-only cookie, session data kept server-side, in a control-plane `app.sqlite3` separate from the monthly message-index DBs). Login rate-limiting/backoff on repeated failures (in-memory sliding window per client IP, process-local by design). CSRF protection on state-changing endpoints (retention settings, logout) via a per-session token. One policy applied consistently regardless of whether accessed via localhost or later via reverse proxy. Default admin password is either `SYLO_ADMIN_PASSWORD` or a randomly generated one logged once on first run -- the installer (section 5) will eventually prompt for it instead, but that's not built yet.
- [x] TLS: not needed for localhost v1; reverse proxy handles TLS termination at scale-out stage (no TLS code added, nothing here to preclude a reverse proxy later)
- [x] Input validation on receiver side against malformed/oversized/malicious syslog payloads (parser must never crash on garbage input) -- covered by section 1's tolerant parser (see receiver spec + parser fuzz-style tests)

## 8. Linux / systemd deployment spec
- [x] Service units: `deploy/systemd/sylo-receiver.service`, `sylo-webapp.service`, `sylo-retention.service`, each invoking the existing `python -m sylo.receiver.main` / `sylo.webapp.main` / `sylo.retention.main` entry points directly via the venv's absolute `python` path. No wrapper class needed here, unlike `winservice.py` -- systemd's own process supervision (`Restart=on-failure`, journal logging via stdout/stderr, captured automatically -- no console-visibility gap like the Windows service's generated-admin-password log line) replaces pywin32's `ServiceFramework`. Retention resolved to the long-lived-loop model, not a `.timer`+`--once` unit -- see open decision 6 below.
- [x] Privileged port 514: UDP/TCP 514 need `CAP_NET_BIND_SERVICE` (or root) on Linux -- Windows has no low-port restriction, which is why this never came up there. `sylo-receiver.service` grants `AmbientCapabilities=CAP_NET_BIND_SERVICE`/`CapabilityBoundingSet=CAP_NET_BIND_SERVICE` and runs as an unprivileged static `sylo` system user (created by `make install` via `useradd --system`, not `DynamicUser=` -- chosen since all three services share the same on-disk `/var/lib/sylo/data` tree, and a shared static UID is simpler to reason about for file ownership than the per-unit ephemeral UIDs `DynamicUser=` hands out).
- [x] Independent start/stop/restart of receiver vs webapp services, mirroring section 1/5's Windows-service independence requirement -- the three units have no `Requires=`/`BindsTo=` between them, so each gets its own `systemctl enable/start/stop/restart` unaffected by the others.
- [x] Default install/data paths on Linux: code+venv under `/opt/sylo` (built by `make install` via `pip install .` into a fresh venv -- not the git clone itself, so the clone directory is disposable after install, matching normal `make install` semantics), data under `/var/lib/sylo/data` (`raw`, `index`, `app.sqlite3`), config at `/etc/sylo/sylo.env` (`EnvironmentFile=` in each unit; `make install` seeds it from `deploy/systemd/sylo.env.example` only if it doesn't already exist, so re-running install to upgrade never clobbers edited config). Distinct locations from the Windows installer's chosen paths, but the same env-var-driven config surface (`SYLO_DATA_DIR`, `SYLO_INDEX_DIR`, `SYLO_APP_DB`, `SYLO_ADMIN_PASSWORD`) just pointed elsewhere, so no code changes were needed to relocate.
- [x] No external runtime dependencies beyond a system Python 3.10+ -- SQLite is stdlib (`sqlite3`), no separate DB server to install; matches section 5's "no external runtime dependencies" goal, just via a venv instead of PyInstaller-bundled exes.
- [x] Packaging format for v1: plain systemd units + a venv/pip install via `make install`/`make uninstall`/`make purge` (see repo-root `Makefile`) -- not a `.deb`/`.rpm`/Docker image, resolving open decision 5 below. `make uninstall` stops services and removes `/opt/sylo` but keeps data+config (mirrors the Windows installer's keep-data-by-default uninstall, section 5); `make purge` also removes `/var/lib/sylo`, `/etc/sylo`, and the `sylo` user for a fully clean removal. Verified: `pip install .` into a fresh venv succeeds and produces cleanly importable `sylo.receiver.main`/`sylo.webapp.main`/`sylo.retention.main` modules, `make -n install`/`uninstall`/`purge` expand correctly, `systemd-analyze verify` accepts all three unit files, and (this pass) a real `sudo make install` + `systemctl start` cycle on a real machine (socrates4) -- which surfaced and fixed a real bug: `pyproject.toml` had no `[tool.setuptools.package-data]`, so `pip install .` shipped `.py` files only and silently dropped `sylo/webapp/static/` and `sylo/webapp/templates/` from the installed venv (`sylo-webapp.service` crashed on startup, `RuntimeError: Directory '.../sylo/webapp/static' does not exist` from `StaticFiles.__init__`). Section 5's PyInstaller specs never hit this because they bundle those dirs explicitly as `datas`; plain `pip install` had no equivalent step. Fixed by adding `sylo = ["webapp/static/**/*", "webapp/templates/**/*"]` under `[tool.setuptools.package-data]`; verified by building a wheel and confirming both dirs are present in it and that `create_app()` no longer raises against an installed copy. Receiver/retention services were unaffected (no static/template assets). Still open: full `systemctl enable/start/stop/restart` independence check across all three units and the 514-bind-as-non-root capability check (see section 7's Linux install test item) -- this pass only exercised the webapp crash and its fix.

## 9. Findings from live deployment (2026-07-15, socrates4, 2 real devices)
- [x] Timestamps displayed 2h off from local wall clock, despite server+devices both set to Europe/Amsterdam and NTP-synced. Root cause: `receipt_time` is captured and stored as UTC (`datetime.now(timezone.utc)`, `sylo/receiver/envelope.py`) -- correct, avoids DST ambiguity in stored/indexed data -- but the webapp templates (`_results.html`, `_tail_row.html`) rendered `row.receipt_time` verbatim with no timezone conversion, so the UI showed UTC labeled as if it were local time. Amsterdam is currently UTC+2 (CEST), matching the reported offset exactly. Resolved client-side (chosen over a server-side `SYLO_DISPLAY_TZ` setting): storage/queries stay UTC-only; each timestamp is now rendered as `<time datetime="{utc-iso}">{utc-iso}</time>` (`_results.html`, `_tail_row.html`) and a small vendored script, `sylo/webapp/static/localtime.js` (no external dependency, consistent with section 3's vendored-only goal), rewrites every `time[datetime]` element's text to the browser's local timezone on `DOMContentLoaded` and again on every `htmx:afterSettle` event -- the latter is what makes this work for content that arrives after the initial page load: paginated `/messages/results` swaps and SSE live-tail rows both go through htmx's swap path (confirmed both `htmx.min.js` and `htmx-sse.min.js` fire `htmx:afterSettle`), so a plain run-once-on-load script would miss them. Verified end-to-end: sent a real syslog message through a live receiver, logged into the webapp, and confirmed the server-rendered fallback carries the correct raw UTC value while the browser-side conversion (checked with Node's `Date` under `TZ=Europe/Amsterdam`) correctly shifts a `07:53:52` UTC receipt time to `09:53:52` local (CEST, UTC+2). - [x] Message column shows the entire raw structured-data blob instead of just MSG -- e.g. a systemd-journal-upload-style RFC5424 message from a real device rendered the full `PRIORITY="6" ... MESSAGE="proxy-confd.service: Deactivated successfully." ...] proxy-confd.service: Deactivated successfully.` instead of just the tail. Root cause was in `sylo/parser.py::_parse_syslog`'s RFC5424 branch: it split the header remainder on the *first* space only (`remainder.split(b" ", 1)`) to separate STRUCTURED-DATA from MSG, which only worked when STRUCTURED-DATA was the nil value `-`. A real bracketed `[SD-ID PARAM="value" ...]` element contains internal spaces, so the naive split cut partway through the SD-ELEMENT and leaked the remainder (including the real MSG) into the `message` field verbatim -- polluting not just the UI but the FTS index (section 2) too. Fixed with a real STRUCTURED-DATA parser, `_split_structured_data` in `sylo/parser.py`: walks the bytes tracking quote/escape state (PARAM-VALUE may contain escaped `\"`, `\\`, `\]` per RFC 5424 §6.3) to find where one or more concatenated SD-ELEMENTs actually end, handles the plain nilvalue (`-`) case the same as before, and falls back to keeping everything if an SD-ELEMENT is unterminated (never guesses/drops content on malformed input, consistent with section 1's parser tolerance requirement). Decided (discussed against the next finding) to discard structured-data content entirely rather than capture any of it (e.g. `_MACHINE_ID`) for later use -- device-identity will instead look at using HOSTNAME (see next finding), which every RFC3164/RFC5424 sender provides, unlike journald-specific SD fields. Covered by 4 new tests in `tests/test_parser.py` (journald-style SD with embedded spaces/quotes -- the real reported case, multiple concatenated SD-ELEMENTs, no-trailing-MSG, and unterminated-SD-ELEMENT fallback); verified end-to-end by replaying the exact reported message through a live receiver and confirming the SQLite index's `message` column holds only `proxy-confd.service: Deactivated successfully.` (the raw per-device text file, section 1's source-of-truth, correctly still keeps the full untouched line).
- [x] Per-device log/index files are keyed by source IP (section 1, deliberately chosen over the parsed hostname field so a spoofed/malformed packet can't inject an arbitrary device identity or path-unsafe characters into the file/queue key). IP isn't stable for devices on DHCP. RFC 3164 has no structured-data concept at all, so it can never carry a MACHINE_ID; RFC 5424 only defines generic STRUCTURED-DATA extensibility, and MACHINE_ID itself is a journald/systemd convention riding on top of that (seen in finding 2's `_MACHINE_ID`), not part of either RFC -- so it's only ever available for journald-based senders, never universal. Settled on HOSTNAME instead, since every RFC3164/RFC5424 sender provides it (RFC5424 as a header field, nilvalue `-` allowed).

  Investigating the actual read path changed the scope of the fix: `sylo/webapp/queries.py`'s `/api/messages` search filter already matches on the parsed `host` column, entirely independent of `source_ip` -- so searching by hostname across a device's DHCP-driven IP change already worked before this pass. The one place that did fragment was `list_devices` (the `/devices` page), which grouped by `source_ip`, so a device getting a new IP showed up as a second, disconnected entry with its own message count/last-seen. Raw per-device text files remain organized under `data/raw/<ip>/...` and split into a new directory on IP change same as before; accepted as a cosmetic/organizational cost, not a data-loss or search-correctness one. Covered by 2 new tests in `tests/test_queries.py` (same hostname across two different source_ips in different months collapses into one device entry; a row with no parsed host falls back to grouping by source_ip). Verified end-to-end through the actual `/devices` HTTP route with seeded rows simulating a DHCP IP change: one row for the device, correct summed message count, most-recent IP shown.

- [x] Real upgrade-over-a-running-install attempt (using the `PrepareToInstall` stop-before-copy logic added in the previous pass, see the port/shortcut finding above) failed: Setup reported `sylo-webapp.exe: DeleteFile failed, code 5` (`ERROR_ACCESS_DENIED`) during `[Files]`, while `sylo-receiver.exe`/`sylo-retention.exe` copied fine -- the installer correctly detected it was reinstalling and ran the stop logic, but didn't actually get `sylo-webapp.exe` unlocked in time.

Flagged as a possible future improvement if 10s ever proves insufficient (e.g. a much slower machine, or multiple browser tabs compounding the drain time -- untested, since `timeout_graceful_shutdown` should cap it regardless of connection count).
- [x] Even the bumped 10s `Sleep` (previous finding) still weren't enough: on this Windows dev machine, a real upgrade attempt caught `sylo-webapp.exe: DeleteFile failed, code 5` live, mid-install, and the offending `Setup - Sylo 0.1.0` error dialog had been sitting there long enough that `SyloWebapp`'s `STOP_PENDING`/`CHECKPOINT` had been frozen at `0x1` for an extended period with no forward progress at all -- not "slower than expected," genuinely wedged.

  Root-caused by direct process inspection, not guesswork: `sylo-webapp.exe` is a PyInstaller **onefile** build (`packaging/pyinstaller/webapp.spec`), which at runtime is actually *two* OS processes -- a launcher/bootloader process (the one the SCM directly starts and tracks) and a child worker process (spawned by the bootloader) that holds the real listening socket and does the actual work. By the time Setup's `[Files]` step hit `DeleteFile failed, code 5`, **both** of those processes had already fully exited (confirmed via `Get-Process`/`Get-CimInstance Win32_Process` -- no `sylo-webapp.exe` PID left at all, `Get-NetTCPConnection` showed the port free). Clicking Inno's own built-in "Try again" in the error dialog then succeeded immediately, with zero code change and no further waiting -- proving the lock was real but had nothing to do with the owning process(es) still running. Windows Defender's real-time-protection process (`MsMpEng.exe`) was confirmed running on the machine; a short-lived AV scan of the just-closed 14MB exe is the most plausible holder of that lock, though it wasn't directly instrumented.

  This means a flat `Sleep`, however long, is structurally the wrong tool here: it can't shorten the common case (file frees up in under a second once the process is gone) and can't reliably bound the uncommon one (a scan's duration isn't a constant Setup can know in advance). Fixed by replacing `PrepareToInstall`'s `Sleep(10000)` with a polling loop, `WaitForUnlock` (`sylo.iss`): it checks the *actual* condition -- `IsFileLocked` via `RenameFile(FileName, FileName)`, which requires exclusive access and fails if anything still holds the file open -- every 500ms, for up to 30s per exe (`sylo-receiver.exe`/`sylo-webapp.exe`/`sylo-retention.exe`, all three guarded uniformly even though webapp is the one with the known slow-shutdown path), instead of guessing a single fixed number for every machine and scenario. If the file is somehow still locked once that cap is exhausted, Inno's own interactive Retry/Skip/Abort dialog remains the fallback, unchanged. Verified end-to-end on a real Windows machine: compiled clean via `iscc`, then two consecutive real upgrade-over-a-running-install cycles both completed without the error dialog appearing at all (previously needed a manual "Try again" click every time).
- [x] Follow-on report after the above fix landed: the web UI port silently reverted from a previously-configured `8080` to the shipped default `8514`, and a previously-set admin password "disappeared" after upgrading. Two distinct causes, found by re-reading `sylo.iss`'s wizard-init code side by side:

  **Admin password -- a real, 100%-reproducible code bug.** `ReadExistingWebPort` (carries the port forward across upgrades) has no equivalent for the password: `AdminPasswordPage.Add(...)` was never seeded from the registry, so `AdminPasswordPage.Values[0]` is always `''` on an upgrade (the page itself is skipped per `ShouldSkipPage`, so nothing else ever populates it). `CurStepChanged` treats that blank as "no password wanted" and omits `SYLO_ADMIN_PASSWORD` from the `WebappEnv` array entirely -- and since `InstallService`'s `RegWriteMultiStringValue` call *replaces* the whole `Environment` registry value rather than merging into it, any previously-set password was silently erased on *every* upgrade, not just once. Fixed by factoring `ReadExistingWebPort`'s parsing logic out into a shared `ReadExistingEnvValue(Key, DefaultValue)` helper and adding `ReadExistingAdminPassword` on top of it, then seeding both password-page fields from it in `InitializeWizard` on upgrade, mirroring the port's existing pattern exactly.

  **Port -- not a logic bug, a concurrency hazard.** Re-reading `ReadExistingWebPort`/`IsUpgrade`/`ShouldSkipPage` found the carry-forward logic itself correct, and this was confirmed empirically: the registry held `8514` steady across two further successful reinstalls after the initial revert, i.e. carry-forward *was* preserving whatever value was currently there. The original 8080-to-8514 flip lined up with two `sylo-setup.exe` processes observed running concurrently on the same machine during testing -- nothing in the script prevented that, and one instance reading "no prior install" (because the other had already stopped+unregistered the old service mid-upgrade) while the first was still mid-flight is a plausible, sufficient explanation on its own. ## 10. Multi-user / remote viewing spec (planned, not yet implemented)
Discussed 2026-07-15: same single install (up to 100 devices), but supporting viewers logging in from other systems on the network, plus a real non-admin role now that more than one person will use it. No per-user device restrictions needed -- any logged-in user (admin or viewer) can see all devices/messages, same as today.

- [ ] **Remote access**: plain HTTP on the LAN, deliberately (user's explicit choice over built-in TLS or a required reverse proxy, aware that session cookie + login password travel unencrypted in transit -- acceptable only because the LAN is trusted). `bind_host`/`SYLO_WEB_BIND_HOST` (`sylo/webapp/config.py`) is already configurable and defaults to `127.0.0.1` -- that default stays as-is (an install shouldn't silently become network-exposed), the deployer opts in explicitly. - [ ] **Roles**: exactly two -- `admin` and `viewer`. No migration system exists today (`appdb.init_db` only does idempotent `CREATE TABLE IF NOT EXISTS`, confirmed via code search, no `migrations/` folder or schema-version table anywhere) -- adding `role`/`active` columns to the `users` table needs a first precedent: a guarded `ALTER TABLE users ADD COLUMN ...` that swallows the "duplicate column" error, run unconditionally on every `init_db` call so it's safe against an already-migrated DB. Since no code today ever creates a second user (confirmed -- only `ensure_default_admin`'s bootstrap-if-`count_users()==0` exists), the migration can unconditionally set `role='admin'` on whatever row(s) already exist before the new-row default becomes `'viewer'`.
- [ ] **Session role checks read fresh, not cached**: `Session` (`auth.py`) gains `role`/`is_active`, looked up from `users` on every `get_session` call (join against `sessions`) rather than stored redundantly on the session row at login time -- same "read settings fresh every time" philosophy already used for `retention_days` (section 4). - [ ] **Retention settings become admin-only**: `/settings/retention` (`routes/settings.py`) currently has *no* role check at all -- confirmed, "logged in" is the only gate today, since there's only ever been one (admin) user. Needs `require_admin` added now that a non-admin role exists.
- [ ] **User management** (new `routes/users.py`, admin-only, `/settings/users`): list users; add a user (username, role, initial password); reset any user's password; deactivate (blocks login + immediately kills that user's existing sessions, per the fresh-read design above) and, separately, delete (only permitted once already deactivated, as a safety rail against instantly deleting an active account). Guard against an admin deactivating/deleting themselves if they're the last remaining admin, same defense-in-depth style as retention's "never delete the current partition" checks (section 4).

- [ ] **Password reveal, once**: no new storage anywhere -- creating/resetting a password just echoes the just-submitted plaintext back once in that response's own HTML (a "Password set for X: ****** [Copy]" panel, `navigator.clipboard`-based copy button, vendored JS in the same style as the existing `localtime.js`, section 9 finding 1). Nothing retrievable after that page is left; only the bcrypt hash persists. Admin types the password; an optional client-side "Generate" button (`crypto.getRandomValues`) is a nice-to-have, not load-bearing.
- [ ] **Templates**: `session` is already passed into every route's template context (confirmed convention across `messages.py`/`devices.py`/`help.py`/`settings.py`) -- gating the "Retention"/"Users" nav links in `base.html` on `session.is_admin` is a small conditional, no new plumbing needed.

## 11. Receiver failure detection / graceful degradation
Prompted by 2026-07-16's live finding (section 9): a corporate EDR product (Cortex XDR) silently blocked the receiver from binding UDP/TCP 514 on a real Windows deployment. Today, a startup failure like that propagates as an unhandled exception and kills the process (confirmed via that finding's event-log traceback) -- on Windows this happens to just stay stopped (no recovery actions configured by default), but on Linux `sylo-receiver.service` has `Restart=on-failure` (section 8), which would crash-loop against a persistent failure like this until systemd's start-limit kicks in. Worse, nothing surfaces this to a person looking at the webapp -- the UI would just quietly show no new messages arriving, indistinguishable from "no devices are sending anything right now."

Goal: the receiver should fail gracefully (no crash-loop, self-heals once the underlying cause is fixed) and the webapp -- a separate process that, per section 3, only ever touches SQLite/files, never the receiver directly -- should be able to detect and display that recording isn't happening.

- [x] **Receiver never exits on a startup/bind failure**: `sylo/receiver/main.py::_start_with_retry` wraps `SyslogServer.start()` in a loop instead of letting the exception propagate through `winservice.py`/`main()` and kill the process -- a fresh `SyslogServer` instance is constructed per attempt (not reused across retries) specifically because a partially-succeeded `start()` leaves its `ThreadPoolExecutor` alive but its indexer/registry half-initialized, and that executor gets `shutdown()`ed during the cleanup `server.stop()` call on the way to retrying, so reusing the same instance on the next attempt would raise `RuntimeError: cannot schedule new futures after shutdown`. On failure: log it, record the error (`ReceiverHealth.mark_failed`), sleep 30s (`_RETRY_INTERVAL_SECONDS`, fixed rather than exponential-with-cap for now -- simplest thing that works, revisit only if retry/log noise over a multi-day outage proves it matters), then retry -- interruptibly (`_interruptible_sleep`, an `asyncio.wait_for(stop_event.wait(), timeout=...)`), so shutdown during a backoff wait is still prompt. Existing supervisor recovery (Windows SCM, systemd `Restart=`) is untouched and remains the safety net for genuinely unexpected crashes elsewhere, not this failure class anymore. Covered by `tests/test_receiver_startup_retry.py::test_run_retries_bind_failure_and_recovers` (occupies the UDP port with a raw socket first, confirms the `run()` task stays alive and reports `state="error"`, then frees the port and confirms it recovers to `state="running"` on its own, no restart).
- [x] **Status file as the cross-process channel**: `sylo/receiver/health.py`'s `ReceiverHealth`, writing atomically (temp file + `rename`) to `<SYLO_INDEX_DIR>/receiver_status.json`. Fields as planned (`state`, `updated_at`, `error`, `error_since`), plus a periodic heartbeat while running (`main.py::_heartbeat_loop`, every 10s) independent of state transitions, so a hard-killed process (no clean stop) is still caught via staleness. `error_since` is held steady across repeated identical failures and only moves on a genuinely *different* error message or a recovery -- verified with a deterministic fake clock (`tests/test_receiver_startup_retry.py`) rather than real wall-clock deltas, after an early version of that same test turned out flaky: two `datetime.now()` calls microseconds apart landed in the same Windows clock tick and compared equal, which a real 1ms `time.sleep` wouldn't have reliably fixed either.
- [x] **Webapp reads and surfaces it**: `sylo/webapp/receiver_health.py::read_receiver_health`, registered as a Jinja global (`app.state.templates.env.globals["receiver_health"]` in `app.py`) rather than added to every route's context dict individually -- `base.html` calls it directly, so the banner applies uniformly with no per-route plumbing. Returns `healthy` plus a separate `reason`/`since` (not one pre-formatted string) specifically so `since` can be rendered as `<time datetime="...">` and get the existing UTC-to-local conversion (`localtime.js`, section 9 finding 1) for free -- confirmed in a live browser check that a UTC `18:49:08` timestamp correctly displays as the same local wall-clock value the rest of the UI uses. Tolerant of a missing file (receiver never started), malformed JSON, and a missing/malformed `updated_at` field, each verified by its own test in `tests/test_receiver_health_reader.py` -- catching an early bug in the process, where a plain `"not json"` file was misreported with the "hasn't reported in yet" wording instead of "malformed" because both cases were sharing one `except` clause.
